# ADR-007: Monorepo Scaling Threshold

## Status
Accepted — 2026-06-03

## Date
2026-06-03

## Context

Astra 采用 Go Monorepo + go.work 管理多个独立版本化子模块。截至 2026-06-03，核心模块已从 47 个优化至 30 个，项目结构趋于稳定。需要明确 Monorepo 的规模上限，避免在团队和代码量增长后出现不可逆的工程效率退化。

### 当前指标

| 指标 | 数值 |
|------|------|
| 核心子模块（go.work） | 30 个（不含 examples/e2e） |
| 总代码行数（.go） | ~45K |
| 全量 CI 时间 | ~20 分钟 |
| 活跃贡献者 | < 10 人 |
| 仓库大小 | ~50 MB |

### 已知的 Monorepo 痛点

**小规模（< 30 模块，< 10 人）：当前阶段**
- ✅ 跨模块重构方便（一次 PR 搞定）
- ✅ 统一版本发布（lockstep tag）
- ✅ 共享 CI 基础设施
- ⚠️ 全量 CI 时间随模块数线性增长

**中规模（30-60 模块，10-30 人）：可预见问题**
- ❌ 全量 CI 时间 > 30 分钟，影响 PR 合并速度
- ❌ go.work 依赖图复杂，`go mod tidy` 耗时显著增加
- ❌ 无关模块的变更触发不相关模块的 CI
- ❌ 语义版本冲突风险上升（模块 A 需要 v2，模块 B 还在 v1）

**大规模（> 60 模块，> 30 人）：业界经验**
- ❌ 仓库克隆/检出时间 > 5 分钟
- ❌ IDE 索引性能退化
- ❌ 依赖管理成为全职工作
- 参考：Kubernetes（~200 模块）已拆分出多个独立仓库

### Go Monorepo 的特殊限制

1. **go.work 不支持条件排除**：无法按团队/目录选择性同步部分模块
2. **语义版本独立化成本高**：每个模块打 tag 需要精确匹配 import path
3. **replace 指令爆炸**：当前已有 25+ 条 replace，每新增模块 +1
4. **无 Bazel/Ninja 等增量构建原生支持**：Go build cache 缓解但不解决全量 CI

## Decision

### 1. 规模阈值

| 阈值 | 触发条件 | 动作 |
|------|----------|------|
| 🟢 健康 | 模块 < 35, CI < 20 min | 维持现状 |
| 🟡 预警 | 模块 35-40, CI 20-30 min | 启动模块合并或拆分评估 |
| 🔴 上限 | 模块 > 40, CI > 30 min | 必须合并/拆分，降低至阈值以下 |

**核心上限**：**40 个子模块**（与 ADR-005 一致），**CI 30 分钟**。

### 2. 规模控制策略

当触及 🟡 预警线时，按优先级执行：

**P0: 继续合并低价值模块**
- 同类功能模块合并（复用 MQ/Discovery/Config 模式）
- 子包扁平化（复用 Runner/TaskQueue/Notify 模式）
- 纯工具模块合入 testutil

**P1: CI 增量优化**
- 按模块变更路径触发部分 CI（path-based CI triggers）
- 缓存 go build cache 跨 PR 复用
- 测试分片并行执行

**P2: 拆分为多仓库**
- 仅当合并和 CI 优化无法满足阈值时启动
- 优先拆出边界清晰的独立项目（如 examples → 独立仓库）
- 保留核心 + 基础设施模块在主仓库

### 3. 拆分判断标准

满足 **以下 3 项中 2 项** 时，模块应拆出为独立仓库：

| 标准 | 定义 |
|------|------|
| 独立发布节奏 | 模块版本与核心无耦合，可独立发版 |
| 独立贡献者 | 模块有 > 3 个非核心团队成员贡献 |
| 独立依赖域 | 模块的第三方依赖与核心无交集 |

### 4. 监控指标

每季度评估以下指标，纳入架构适应度函数：

```yaml
monorepo_scaling:
  module_count:
    green: 35
    yellow: 40
    red: 45
  ci_full_duration_min:
    green: 20
    yellow: 30
    red: 40
  repo_size_mb:
    green: 100
    yellow: 200
    red: 500
  active_contributors:
    note: "当 > 20 人时需重新评估 Monorepo 适用性"
```

## Options Considered

### Option A: 严格 Monorepo（不设上限）
- **Pros**: 最大化跨模块重构能力
- **Cons**: 不可持续，业界经验表明 50+ 模块后效率急剧下降

### Option B: 立即拆分为多仓库
- **Pros**: 各仓库独立演进，CI 隔离
- **Cons**: 跨仓库重构困难，版本兼容性管理成本高，当前团队规模不需要

### Option C: 阈值驱动演进（选定方案）
- **Pros**: 在 Monorepo 效率下降前预警，给团队充足决策时间
- **Cons**: 阈值需要根据实际经验调优

## Consequences

### 得到
- **明确的规模红线**：不会在不知不觉中超过可管理规模
- **渐进决策框架**：不急于拆分，但在需要时有清晰路径
- **量化监控**：架构适应度函数自动检测规模退化

### 放弃
- **无限制增长**：新增模块必须经过合并/拆分评估
- **Monorepo 的绝对便利**：达到阈值后部分模块需接受多仓库管理的代价

### 对当前项目的影响

当前 30 个核心模块远低于 🟢 健康线（35），无需立即行动。
已完成的模块合并（47→30）为未来增长预留了 5 个模块的缓冲空间。

新模块引入流程建议：
1. 评估是否可与现有模块合并（优先）
2. 无法合并时，检查是否超过 🟡 预警线
3. 超过预警线时，先合并/扁平化其他模块腾出空间
4. 超过 🔴 上限时，启动拆分评估
