## 🔧 技术决策记录

### 决策 1: 为什么使用 Gin 而不是 Astra 内置的 HTTP 服务器？

**选择**: 使用 Gin 作为 HTTP 框架（Astra 可以集成 Gin）

**原因**：
1. Gin 有更好的文档和社区支持
2. Gin 的中间件生态系统更丰富
3. 团队对 Gin 更熟悉
4. Astra 的定位是"增强 Gin"，而非替代

**权衡**：
- ✅ 开发速度快
- ✅ 学习曲线平缓
- ❌ 未完全展示 Astra 的 HTTP 能力

---

### 决策 2: 为什么使用 GORM 而不是 database/sql？

**选择**: 使用 GORM 作为 ORM 库

**原因**：
1. 路线图要求"ORM + 缓存"
2. GORM 支持 AutoMigrate（快速原型）
3. GORM 有 Hook 机制（适合缓存失效）
4. 团队对 GORM 更熟悉

**权衡**：
- ✅ 开发效率高
- ✅ 数据库抽象好
- ❌ 性能略低于 database/sql（可接受）

---

### 决策 3: 为什么使用 gRPC 而不是 REST 用于微服务通信？

**选择**: 使用 gRPC 作为评论服务的通信协议

**原因**：
1. 路线图要求"gRPC 服务间调用"
2. gRPC 性能高（Protobuf 序列化）
3. 强类型接口定义（.proto 文件）
4. 支持流式通信（未来扩展）

**权衡**：
- ✅ 性能好
- ✅ 接口清晰
- ❌ 调试略困难（需要 gRPC UI）

---

### 决策 4: 为什么使用 Kafka 而不是 RabbitMQ？

**选择**: 使用 Kafka 作为消息队列

**原因**：
1. 路线图要求"Kafka（异步通知）"
2. Kafka 吞吐量高（适合通知场景）
3. Kafka 有持久化（不会丢失消息）
4. 团队有 Kafka 经验

**权衡**：
- ✅ 可靠性高
- ✅ 扩展性好
- ❌ 部署复杂度高（需要 Zookeeper）

## 📝 关键学习点

### 1. 分层架构的好处

**问题**: 为什么不直接在 handler 中写 SQL？

**答案**: 分层架构（domain → repository → service → handler）有以下好处：
- **可测试性**: 可以 mock repository 测试 service，mock service 测试 handler
- **可维护性**: 修改数据库表结构只需修改 repository，不影响 service 和 handler
- **可扩展性**: 可以添加缓存层（在 service 中检查 Redis）而不影响 repository

---

### 2. gRPC  vs REST

**问题**: 为什么评论服务用 gRPC，而 API 服务器用 REST？

**答案**: 
- **外部客户端**（浏览器、移动端）更适合 REST（HTTP/JSON 方便调试）
- **内部微服务通信**更适合 gRPC（性能高、强类型、支持流式）

**实践经验**:
- gRPC 适合 "server-to-server" 通信
- REST 适合 "client-to-server" 通信

---

### 3. Kafka 异步通知

**问题**: 为什么通知要用 Kafka 异步处理？

**答案**:
- **解耦**: API 服务器不需要等待邮件发送完成
- **可靠性**: 如果邮件发送失败，Kafka 可以重试
- **扩展性**: 可以添加多个 Worker 消费同一个 Topic

**实践经验**:
- 通知类型：评论通知、回复通知、提及通知
- Kafka Topic 设计：`blog-notifications`, `blog-comments`
